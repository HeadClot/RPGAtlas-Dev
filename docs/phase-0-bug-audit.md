**Date:** 2026-07-01
**Scope:** `js/editor.js`, `js/engine.js`, `js/data.js`, `js/assets.js`, `js/renderer.js`,
`js/sfx.js`, `js/plugins.js`, `js/quests.js`, `js/journal-view.js`,
`js/runtime/{input,messages}.js`, `js/editor/{project-io,host,i18n}.js`,
`src-tauri/src/{lib,main}.rs`. Read-only audit — no source files were modified.

**GitHub issues:** `gh` CLI is not installed in this environment (`gh: command not found`,
exit 127), so `gh issue list` could not be run. Open issues on
`https://github.com/DriftwoodGaming/RPGAtlas` were not reviewed as part of this pass;

## Summary

The codebase is in noticeably good shape for a pre-Phase-0 monolith: there are **zero**
TODO/FIXME/HACK comments anywhere in `js/`, error handling in the plugin runtime, i18n, and
Tauri Rust backend is consistently correct, and several areas that looked suspicious at a
glance (self-switch scoping, erase-during-execution, playtest handshake races, resizeMap
bounds math) turned out to be handled correctly on closer reading — those are listed under
"not bugs" below so they aren't re-litigated later. The real risk is concentrated in three
places: **battle state consistency** (an enemy-array index mismatch and a stale dead-ally
heal target, both of which can throw or silently break battle logic), **save/load
robustness** (an in-game save-to-slot write with zero error handling, and a save/load path
that throws an uncaught exception if the project's maps change between saves — directly
reachable by deleting a map that a save file, start-map setting, or transfer command still
references), and **long-session hygiene** (a real stuck-key bug on focus loss, a WebGL
context-loss path that never recovers until the next map transfer, and a GPU texture leak on
custom-character edits). None of these are exotic to trigger — the map-delete/save-crash and
stuck-key bugs in particular are things a normal playtest session will hit. Findings below
are only the ones verified by reading the actual control flow (not pattern-matched); a few
plausible-looking leads (an interpreter "soft-lock" via self-switches, a stale-page crash)
were investigated and ruled out after tracing the exact execution order — see "not bugs"
for why.

**Totals:** 4 Critical, 7 High, 8 Medium, 5 Low = 24 findings.

## Not bugs (investigated and ruled out)

- **Interpreter "soft-lock" via self-switch mid-execution** — hypothesized that a command flipping a self-switch to deactivate the currently-running event's own page would null `rt.page` while `rt.kind` stayed `"human"`, crashing `runEventBlocking`'s `finally` block and permanently wedging `blockingRun`. Traced in detail: `refreshPage()` always updates `rt.kind` and `rt.page` atomically in the same call, so `rt.kind` can never be `"human"` while `rt.page` is `null` by the time the `finally` block runs. Not reproducible as hypothesized.
- **Self-switch scoping** — correctly namespaced as `mapId:eventId:switchKey` everywhere (engine.js:506, 782, 1031, 1266); no cross-event or cross-map leakage.
- **Erase-event during its own execution** — `case "erase"` only sets an `erased` flag (gates rendering/collision); the interpreter's command-array reference stays valid regardless, so the rest of the event's own command list still runs safely to completion. Matches standard RPG-maker semantics, not a bug.
- **Playtest window handshake race** — there is no `postMessage`/`BroadcastChannel` handshake at all; both the web path (`saveNow()` then synchronous `localStorage.setItem` then `window.open`) and the Tauri path (`saveNow()` then `invoke("open_playtest")`, which reloads a pre-existing hidden window) save synchronously before opening/reloading the play window, so there's no read-before-write race in the code as written.
- **Map resize tile/height/shadow remapping** — bounds-checked correctly with `Math.min(oldDim, newDim)`; growth fills new cells with explicit defaults, not `undefined`. Only the event-deletion side-effect (see Medium) lacks a safety net.
- **Plugin runtime error isolation** — verified correct: `Plugins.fire`/`fireRender` wrap each hook in try/catch and self-disable the offending plugin by splicing it out rather than crashing the engine or other plugins; plugin load errors are captured per-plugin and surfaced via `window.AtlasPluginStatus`.
- **Gamepad disconnect handling** — `pollGamepads()`/`reseedPadHeld()` both guard `navigator.getGamepads()` in try/catch and clean up stale slots by diffing against currently-seen indices every poll, independent of whether the `gamepaddisconnected` event fires.
- **i18n missing-key fallback** — `t()` falls back to the raw source string, never throws or renders `"undefined"`.
- **Tauri Rust backend** — no `.unwrap()`/panic-risk in any `#[tauri::command]` handler; all file I/O and dialog errors map to `Result<_, String>` and propagate to JS cleanly. The only `.expect()` in the crate is the top-level `Builder::run()` call, a legitimate fatal-at-startup case.
- **Schema version comparison** — `Number(p.meta.formatVersion) || 0` is a proper numeric coercion, not a string compare; missing/undefined versions correctly treat as `0`.
- **Async texture-upload races on map transition** — `Renderer.setMap` is synchronous end-to-end (canvas `drawImage` + `gl.texImage2D`, no `Image.onload`/`fetch` inside it), and `engine.js`'s `loadMap`/`prerenderMap` fully await each step, so a fast double map-transfer cannot interleave a stale texture upload.
- **sfx.js audio lifecycle** — oscillators/buffer sources are standard fire-and-forget WebAudio nodes eligible for GC once disconnected; `Music.stop()` is always called before a new interval is armed, so no duplicate-timer accumulation on rapid theme switching.

## Debt (not bugs, but worth planning for)

- No save-file format version field exists anywhere in `engine.js`'s save payload — there is currently no way to detect or migrate an older/foreign save shape (contrast with `data.js`'s project `formatVersion` + migration registry). Worth adding alongside the Phase 1 typed-schema work, since it's the same shape of problem.
- `js/data.js`'s single migration step (`_migrateV0toV1`) is not internally idempotent (re-invoking it on already-migrated data could re-push duplicate plugin metadata mutations); currently safe only because `migrateProject` never re-runs a completed step. This is a latent trap for whoever writes migration step 2 — worth a guard or a comment now, before Phase 1 needs to add one.
- The quest-objective dangling-reference validator (`editor.js:4020-4038`) has no equivalent for maps (start map / transfer targets) or, likely, other cross-referencing ids elsewhere in the project — a general "reference integrity" pass would be a natural Phase 1/3 database-upgrade item.
- `js/quests.js` and `js/journal-view.js` are clean (no TODOs, no swallowed errors, consistent guard patterns) — nothing to carry forward from this audit for those two files.
